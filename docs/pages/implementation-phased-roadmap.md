# 実装ロードマップ（Phase分割）

このドキュメントは、`docs/pages/README.md` を実装に落とすための進行表です。  
方針は「各フェーズで **CursorのPLAN -> 実装** を繰り返す」です。

---

## 進め方の基本ルール
- 1フェーズずつ完了させる（並行で広げすぎない）
- 各フェーズ開始時に、下記の「PLAN作成依頼テンプレ」をそのままCursorに渡す
- フェーズ完了時に「完了条件」を満たしたかをチェックしてから次へ進む
- モック前提でも、`状態の受け渡し` と `効果KPI` は必ず実装に入れる
- 固有名詞を使う場合は `data-redaction-dictionary.md` の命名規約に従う

---

## Phase 0: 実装基盤の固定

### 目的
- 実装の土台（共通state、KPI辞書、演出state）を先に固定し、後工程の手戻りを防ぐ。

### 具体指示
- 共通状態キーを型として定義する（READMEのKPI辞書/状態受け渡し契約に準拠）
- UI演出stateを統一: `idle/loading/success/retry`
- グローバルなデモ状態管理（Context or store）を導入
- トースト/ローディング表現の共通ユーティリティを作る

### 完了条件
- 主要ページから共通stateを参照できる
- 演出stateが共通型で扱える
- KPIキーが1箇所で定義されている

### 次フェーズへ渡すもの
- `state schema`（型）
- `UI演出ヘルパー`
- `KPI辞書`

### Cursor PLAN作成依頼テンプレ
```text
Phase 0を実装したいです。対象は共通実装基盤のみです。

目的:
- 共通state、演出state、KPI辞書を先に固定して後続フェーズの手戻りを防ぐ。

必須実装:
- 状態受け渡しキー（selectedCandidateId など）の型定義
- idle/loading/success/retry の共通化
- KPI辞書（timeSavedMinutesPerDay など）の共通化
- 全ページで使える最小の状態管理レイヤー

制約:
- DB/API連携は行わない
- モックデータ前提

完了条件:
- home, candidate-detail, knowledge の3ページから共通state参照が可能
- KPIキーが1箇所で管理されている
```

## Phase 1: コア動線（home -> candidate-detail -> knowledge）

### 目的
- 3大機能の価値を最短で体験できる「勝ち筋導線」を先に完成させる。

### 具体指示
- `home` に業務完了導線を実装（今日フォロー対象の完了アクション）
- `candidate-detail` にAI爆速プロファイリング演出を実装
- `knowledge` に学習ログ反映演出を実装
- 3ページ間で状態受け渡し（候補者ID、提案先、ナレッジ反映版）を接続

### 完了条件
- 3ページを順に操作すると、状態が連続して見える
- 「査定 -> 行動 -> 学習」の一連ストーリーが3分以内でデモ可能

### 次フェーズへ渡すもの
- コア導線の遷移仕様
- 状態受け渡し実績（どのキーが使われるか）
- デモ台本（短尺版）

### Cursor PLAN作成依頼テンプレ
```text
Phase 1を実装したいです。対象は home -> candidate-detail -> knowledge の3ページです。

目的:
- 3大機能（査定高速化・営業司令塔・ナレッジ資産化）を最短導線で成立させる。

必須実装:
- home: 今日フォロー対象の完了アクションと次導線
- candidate-detail: PDF査定演出（段階ローディング、結果表示、次アクション）
- knowledge: メモ保存 -> 学習反映の演出
- 3ページ間の状態受け渡し接続

制約:
- 実API接続なし、モック完結

完了条件:
- 3ページ連続デモで状態が自然に引き継がれる
```

## Phase 2: 業務導線（candidates / matching / clients / client-detail / documents）

### 目的
- 実運用で「本当に便利」と感じる候補者提案導線を完成させる。

### 具体指示
- `candidates`: 絞り込み -> 優先候補選定 -> 詳細遷移
- `matching`: 提案候補確定と理由提示
- `clients` / `client-detail`: 案件選定と適合候補の提示
- `documents`: 不備解消フローとパイプライン復帰導線
- 各ページに「業務完了導線」イベントを実装（完了ログ化）

### 完了条件
- 候補者選定から提案確定まで迷わず流せる
- 書類不備が提案遅延に効くことを画面で示せる

### 次フェーズへ渡すもの
- 提案導線の完了イベント一覧
- 不備/復帰の業務シナリオ

### Cursor PLAN作成依頼テンプレ
```text
Phase 2を実装したいです。対象は candidates/matching/clients/client-detail/documents です。

目的:
- 候補者選定から提案確定までを業務導線として完成させる。

必須実装:
- 各ページの業務完了導線（完了イベント）を実装
- 状態受け渡し（candidate/client/match/document）を接続
- documentsで不備解消 -> 候補者進行復帰の演出

制約:
- DB連携なし、モックデータのみ

完了条件:
- 「選定 -> 査定 -> 提案 -> 書類復帰」まで一気通貫で操作可能
```

## Phase 3: 補助導線（operations / revenue / messages / field-reports / more）

### 目的
- 「導入後の運用イメージ」を補強し、デモの商談説得力を上げる。

### 具体指示
- `operations`: 日次運用ハブと未処理タスク導線
- `revenue`: 経営言語KPIの集約表示
- `messages`: 翻訳と優先度判定導線
- `field-reports`: 現場報告完了導線
- `more`: モバイルショートカット最適化

### 完了条件
- コア導線を邪魔せず、必要なときに補助ページで価値を補強できる
- 経営層向けにROI説明が可能

### 次フェーズへ渡すもの
- 経営向け説明キャプチャ/デモシナリオ
- 現場運用向け説明ポイント

### Cursor PLAN作成依頼テンプレ
```text
Phase 3を実装したいです。対象は operations/revenue/messages/field-reports/more です。

目的:
- 導入後の運用とROIを示す補助導線を完成させる。

必須実装:
- operations: 日次オペの健全性導線
- revenue: 経営言語KPIの見せ方強化
- messages/field-reports/more: 現場実務の補助導線

制約:
- コア導線の邪魔をしない
- モック完結

完了条件:
- 現場目線と経営目線の両方で価値説明ができる
```

## Phase 4: 商談最適化（デモ専用仕上げ）

### 目的
- 「便利そう」ではなく「導入したい」に変える最終調整。

### 具体指示
- 全ページのマイクロコピーを統一（学習/同期/完了のトーン）
- 主要導線の遷移時間とアニメーション体験を最適化
- 疑似障害 -> 再試行復帰の演出を1箇所入れる
- 3分版・10分版のデモ台本を作る

### 完了条件
- 3分デモで価値が伝わる
- 10分デモで導入イメージまで説明できる
- 操作につまずく箇所がない

### Cursor PLAN作成依頼テンプレ
```text
Phase 4を実装したいです。対象は商談デモ最適化です。

目的:
- 3分/10分の商談デモで「導入価値」を明確に伝える。

必須実装:
- マイクロコピー統一
- 主要導線の体験最適化（速度、アニメーション、トースト）
- 疑似障害 -> 再試行復帰のデモ演出
- デモ台本（3分版/10分版）をドキュメント化

完了条件:
- デモ担当者が台本どおりに再現可能
```
