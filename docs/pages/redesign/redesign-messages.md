# redesign `/messages`

## ページの意思決定目的
受信メッセージの緊急度を判断し、対応優先順を決める。

## 現状課題
- 翻訳表示が主目的化し、対応判断が後回しになる。
- 重要メッセージの優先順位が視覚的に弱い。
- 対応後の状態更新が分かりにくい。

## 新情報設計（主役/準主役/補助）
- 主役: 緊急対応キュー（危険/警告/通常）。
- 準主役: 原文/訳文/要約と推奨対応。
- 補助: 既読履歴、エスカレーション履歴。

## レイアウト仕様（PC/Tablet/Mobile）
- PC: 左キュー、右詳細（原文/訳文/推奨対応）。
- Tablet: 上段キュー、下段詳細。
- Mobile: キュー選択式、詳細は全画面シート。

## 主要コンポーネント差し替え案
- `MessagePriorityQueue`（新規）: 優先度順リスト。
- `TranslationDecisionPanel`（新規）: 翻訳と対応提案を統合。
- `ResolveActionBar`（新規）: 対応完了/保留/エスカレーション。

## マイクロコピー置換表
- AI翻訳を表示 -> 訳文を表示して対応を判断する
- 要注意 -> 優先対応
- 確認済み -> 対応方針を記録済み

## 状態演出（idle/loading/success/retry）
- idle: 未対応件数と危険件数を表示。
- loading: 「翻訳中 -> 緊急度判定中」。
- success: 対応済み行を下段へ移動。
- retry: 翻訳失敗時は要約のみ表示しエスカレーション導線を表示。

## 遷移・状態受け渡し変更点
- 候補者確認: `/candidates/[id]` へ `messageRiskLevel` を渡す。
- 実務連携: `/operations` へ `messageResolvedAt` を渡す。

## 実装対象ファイル
- `app/messages/page.tsx`
- `lib/demo-messages.ts`
- `lib/demo-state/types.ts`

## 受け入れ基準（操作ベース）
- 45秒以内に優先対応1件を処理できる。
- 処理後にメッセージ状態が更新される。
- 原文が読めなくても行動判断が可能なUIになっている。
