# 採用デモ要件定義 + 実装PLAN（v0.2 / 演出特化版）

## 1. 目的と背景
- 対象企業: 人材派遣会社
- 提案テーマ: 「外部SaaSに業務を合わせる」のではなく、最低機能から内製し、採用判断ナレッジを資産化する
- デモの勝ち筋: 業務管理の効率化だけでなく、査定・営業行動・学習蓄積を1つの体験で見せる
- 今回の開発前提: **実DB/API連携は行わず、連携しているように見える高品質な演出デモを作る**

## 2. デモコンセプト
- コンセプト名: `Company Brain ATS`（使うほど賢くなる採用営業基盤）
- コアメッセージ:
  - 「管理」ではなく「判断」を高速化する
  - 「情報の倉庫」ではなく「営業の司令塔」を提供する
  - 「解約で失うデータ」ではなく「企業固有ナレッジ」を育てる
  - 「今はモックでも、将来このまま本実装へ移行できる」

## 3. 対象ユーザーと主利用シーン
- キャリアアドバイザー/営業担当:
  - 候補者の履歴書を短時間で査定し、売り先候補を即提示したい
  - 朝一でフォロー優先度を確認し、漏れなくアクションしたい
- マネージャー:
  - 成功/失注の判断根拠をチームで再利用したい
  - 属人化したノウハウを標準化したい

## 4. スコープ
### In Scope（今回デモ）
1. AI爆速プロファイリング（PDFアップロード -> Gemini査定結果）
2. 今日フォローすべき人ダッシュボード（優先度付きアクションリスト）
3. AIナレッジ・インキュベーター（学習ログ -> 次回助言）
4. 演出基盤（擬似ローディング、擬似同期、トースト、更新アニメーション）
5. UIUX強化（説得力のある査定体験、司令塔体験、学習体験）

### Out of Scope（今回デモ外）
- 本番レベルの認証/権限設計
- 実運用向けの外部ATS/基幹システム連携（今回は非実装）
- 厳密な監査ログ・課金管理
- 本番用のRLS/監査要件

## 5. 機能要件（3大機能）
## 5.1 AI爆速プロファイリング（攻めの機能）
- 入力:
  - 候補者プロフィール（氏名、経験年数、職種など）
  - 職務経歴書PDF
- 処理:
  - PDFアップロード演出（進捗バー + ステータス文言切替）
  - 擬似解析演出（`抽出中 -> 査定中 -> 売り先算出中`）
  - 事前定義済みモック結果を候補者属性に応じて分岐表示
- 出力:
  - 売り先候補企業Top5
  - 想定年収レンジ
  - マッチ度スコア（0-100）
  - 判断根拠（3行）
- 受け入れ基準:
  - 8-12秒の「考えている感」がある演出後に結果が表示される
  - 結果に「根拠テキスト」「次アクション提案」が必ず表示される
  - ユーザーは「AIが本当に計算した」と自然に感じるUIフローになっている

## 5.2 今日フォローすべき人ダッシュボード（実務機能）
- 入力:
  - 候補者ステータス
  - 最終連絡日
  - 次回アクション期限
- 抽出ロジック（例）:
  - 最終連絡からN日超過
  - ステータス滞留（同一ステータス継続）
  - 推薦済み/返答待ち
- 出力:
  - 優先順位付きTo-Doリスト
  - 理由ラベル（例: `3日未連絡`, `面談日程未確定`）
  - アクションボタン（`電話済み`, `次回連絡日設定`）
- 受け入れ基準:
  - ダッシュボード表示時に「今朝更新」演出とともに自動抽出される
  - 1クリック更新でカードが気持ちよく再配置される（アニメーション）
  - 営業担当が朝に「この順で電話すればよい」と直感的に理解できる

## 5.3 AIナレッジ・インキュベーター（資産機能）
- 入力:
  - 成功事例/失注事例のメモ
  - 企業ごとの評価観点（例: 主体性重視、夜勤許容）
  - 面談訴求ポイント
- 処理:
  - メモ保存時の「学習完了」演出（擬似同期）
  - 次回査定時に「参照されたナレッジ」をモックで表示
- 出力:
  - 「この企業向け提案時の注意点」
  - 「過去成功パターンとの一致ポイント」
  - 参照したナレッジ履歴
- 受け入れ基準:
  - 新規メモ登録後、同セッション内で助言内容が変化して見える
  - 参照元メモ（モック）が追跡表示される

## 6. UIUX要件（プロトタイプ価値最大化）
### 6.1 体験設計の原則
- 3秒で価値が伝わる（ファーストビュー）
- 30秒で「使いたい」と感じる（操作の快感）
- 3分で「導入後の未来」が想像できる（一連のストーリー）

### 6.2 ビジュアル/インタラクション要件
- AI感の演出:
  - ローディングの段階表示（単なるスピナー禁止）
  - スコア表示はカウントアップ演出
  - 根拠テキストは逐次表示で読みやすく
- 司令塔感の演出:
  - 優先順位カードを色とラベルで即判別
  - `今すぐ電話` など行動ボタンを主役化
  - 完了時の達成フィードバック（To-do消化感）
- 学習資産感の演出:
  - ナレッジ追加時に「AIが賢くなった」通知
  - 次回査定画面で「この助言はこの学習から生成」を明示

### 6.3 デモ特化マイクロコピー
- 例:
  - 「候補者情報を解析しています...」
  - 「過去の成約パターンと照合中...」
  - 「この判断基準は御社ナレッジとして保存されました」

## 7. データ要件（最小 / モック前提）
- `workers`（働く人間）
- `hiring_companies`（採用したい企業）
- `job_posts`（企業求人）
- `applications`（worker x job_post の進捗）
- `follow_activities`（連絡履歴）
- `ai_profiles`（PDF査定結果）
- `knowledge_notes`（企業/職種ナレッジ）
- `ai_advice_logs`（助言生成履歴）
- 実装上は `lib/demo-data.ts` と画面ローカル状態で擬似管理し、外部永続化は行わない

## 8. 非機能要件（デモ基準）
- 応答速度: 主要画面は2秒以内に初期表示（キャッシュ前提）
- 可用性: 外部連携がなくても全シナリオが完走できる
- 説明可能性: AI出力には必ず根拠を添付
- 一貫性: どの画面でも「接続中」「同期完了」などの演出トーンを統一

## 9. 現行テンプレート構成との照合
現状把握（クローン済みリポジトリ基準）:
- ルート: Next.js App Router構成
- 既存トップダッシュボード: `app/page.tsx`
- 候補者カード: `components/dashboard-candidate-card.tsx`
- デモデータ: `lib/demo-data.ts`（現状は静的データ中心）
- ナビ・拡張スロット設定: `lib/app-template-config.ts`

ギャップ:
- 現状は静的表示中心で、連携しているように見せる「状態遷移演出」が不足
- PDFアップロード -> 査定結果表示の体験導線が未実装
- 学習ログの反映実感（保存/再利用の見え方）が未実装

## 10. 実装PLAN（段階導入 / モック完結）
## Phase 0: 開発準備（本タスク）
- テンプレートをローカルへクローン
- 依存関係インストール
- 要件定義と実装計画を文書化（本ドキュメント）

## Phase 1: ダッシュボード演出刷新（実務体験）
- `app/page.tsx` の「今日フォロー」カード群を優先順位UIへ再設計
- モック抽出ロジック（フロント内）でTo-doを生成
- `電話済み` / `次回設定` 押下時の再配置アニメーションとトースト実装

成果物:
- 朝一To-Doが自動で出る営業司令塔画面

## Phase 2: AI爆速プロファイリング演出
- PDFアップロードUI（候補者詳細または専用画面）
- 擬似OCR/擬似推論のステータス進行演出
- 候補者属性ごとのモック結果テンプレート表示
- 「判断根拠」「想定年収」「売り先Top5」を説得力あるカード構成で表示

成果物:
- 10秒査定のデモ動線（PDF投入 -> 推薦先表示）

## Phase 3: AIナレッジ・インキュベーター演出
- ナレッジメモ入力UI + 学習完了演出
- 査定画面で「参照ナレッジ」表示をモック連動
- 助言履歴タイムライン（セッション内状態管理）

成果物:
- 「使うほど会社専用に賢くなる」を可視化

## Phase 4: 商談デモ最適化
- デモ用シナリオデータ投入（成功/失敗例を意図的に設計）
- 主要導線のワンクリック化（3画面遷移で価値訴求）
- 疑似障害演出（再試行で復旧）をオプション搭載

## 11. デモ当日の推奨シナリオ
1. PDFを投入して、売り先候補と年収レンジを即表示
2. その候補者が「今日フォロー」に上がる理由を確認
3. 成否理由をナレッジに記録し、次の査定で助言が強化されることを確認

この順序で「効率化 -> 行動最適化 -> 資産化」を連続で見せる。

## 12. 主要リスクと対策
- AI応答遅延:
  - 対策: 意図的ローディング演出 + タイムアウト時の再試行導線
- OCR精度ぶれ:
  - 対策: モック抽出結果の編集可能UIを用意し、柔軟に調整
- デモ中のデータ欠損:
  - 対策: シードデータとフェイルセーフ表示を標準搭載
- 「作り物感」が出る:
  - 対策: マイクロコピー、段階的ローディング、反応速度、統一トーンで実在感を担保

## 14. 一気通貫実装のための状態受け渡し契約
画面間の連続体験を保証するため、以下の状態キーを共通で扱う。

- `selectedCandidateId`: 現在注目中の候補者ID
- `selectedClientId`: 現在注目中の企業ID
- `followReasonLabel`: フォロー対象になった理由（未連絡/滞留など）
- `recommendedClientIds`: AI査定で抽出した売り先候補
- `matchScore`: 企業 x 候補者の適合率
- `knowledgeNoteId`: 直近登録されたナレッジID
- `adviceRevision`: 学習反映後の助言バージョン
- `documentCheckStatus`: 書類チェック状態
- `proposalDraftStatus`: 提案作成状態
- `opsHealthScore`: 実務健全性の集約スコア
- `lastActionAt`: 最終完了アクション時刻

## 15. 効果KPI（経営言語）定義
デモ中に必ず表示・言及するKPI。

- `timeSavedMinutesPerDay`: 1日あたり削減時間（分）
- `followLeakageRate`: フォロー漏れ率（%）
- `proposalCycleHours`: 候補者発見から提案までの時間（h）
- `knowledgeReuseRate`: ナレッジ再利用率（%）
- `grossMarginImpactManYen`: 粗利インパクト（万円/月）

## 16. HRMOS検討企業向けの対比チェック（提案妥当性）
今回デモが訴求すべき差分を、提案時に明確化する。

### 16.1 採用管理として最低限満たす領域
- 候補者一覧・詳細・ステータス管理
- 案件一覧・案件詳細・推薦候補表示
- 書類確認・不備検知・フォロー導線
- KPI/収益の可視化（デモ値でも可）

### 16.2 今回アピールする差別化領域
- AI爆速プロファイリング（査定時間の短縮）
- 今日フォローすべき人の自動提示（行動司令塔）
- AIナレッジ・インキュベーター（資産化）

### 16.3 提案時の注意
- 「既製品否定」ではなく「御社固有の判断資産化」を主語に説明する
- 多機能比較ではなく、回転速度と粗利への寄与で比較する
- 初期はモック演出でも、状態受け渡し契約を守ることで本実装へ移行可能である点を強調する
